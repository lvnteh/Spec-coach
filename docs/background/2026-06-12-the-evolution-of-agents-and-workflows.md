# The evolution of agents and workflows

> *This article is based on a workshop held on June 12, 2026 and was generated by AI from the slides and transcription.*

The agent landscape has churned hard enough over the last three years that the techniques we used at the start are coming back around — re-combined, re-named, but recognisably the same building blocks. The point of this workshop was to walk through that arc end-to-end on a single toy problem and see what changes at each step. By the time we got to the end of the afternoon we had a three-tier setup running locally — an A2A agent in front of an MCP server in front of a REST backend — and the path from there to the production agents we'll actually be building starts to look very short.

The problem itself was deliberately trivial: a personalization scenario with three operations — `registerTemplate`, `compileTemplate`, `send` — wired to a REST backend that ships with the workshop repo. The user types something like "Send a welcome message to Viktor Somodi" and a notification eventually lands on the laptop's notification centre. The whole point of keeping the task this small is so that the only thing that changes across exercises is *how* the agent is built — same backend, same operations, same prompt.

For each implementation we look at the same seven dimensions:

- **Available tools** — what can the agent actually do?
- **Available workflows** — none, hand-rolled in code, in a prompt, in a skill, or as a structured spec?
- **Control** — who decides the next step: code or the LLM?
- **Speed** — how many round-trips before the user sees a result?
- **Reliability** — will it do the same thing twice in a row?
- **Context management** — what stays in the prompt, what gets summarised, what gets dropped?
- **Dynamic** — how easily does the same setup handle a request you didn't think of?

These dimensions are how I think about every agent design now. They make the trade-offs explicit instead of letting each new framework re-introduce them as features.

## The static workflow — where we used to live

Three or four years ago there were no agents and no tools, in the sense we mean today. There was one HTTP call: you handed the model a prompt, you got something back. That was the entire surface area. So if you wanted to do anything more interesting than answer a single question — and almost every real problem is more interesting than that — you decomposed the task into steps, kept the LLM call as narrow as possible (usually "turn this free-text request into a structured object"), and wrote ordinary code around it.

For our personalization example three years ago, the flow would have looked like this. A request comes in. First you ask the LLM: should we even bother with this, or is it off-topic? That returns true or false. If true, you call the LLM again to extract the structured pieces — what's the message, who's it for, which placeholders does it need. The code then validates that the extraction is well-formed, calls `registerTemplate` over HTTP, takes the result and passes it to `compileTemplate`, takes that and passes it to `send`. At the end maybe one more LLM call to format a nice confirmation message. The whole thing is orchestrated by code.

In our seven dimensions, this is unambiguous:

- The "tools" are just functions in your codebase — the LLM never sees them. The code calls them.
- The workflow is code. You wrote it.
- Control sits firmly in the code.
- Speed is the best you'll ever get. The only LLM round-trips are the ones you explicitly decided to make.
- Reliability is the best you'll ever get. The order in which things happen is deterministic.
- Context management is yours to handle. You decide what goes into each LLM call and what comes out.
- Dynamic capability is bad. You can only handle the workflows you wrote code for. If someone asks "what templates do I have?" and you didn't build a workflow for that, the answer is "we don't support that." The way to expand coverage is to write more code.

There are well-known patterns for taking the edge off the rigidity — chain workflows in sequence, run them in parallel, put a router LLM call at the front that decides which sub-workflow to dispatch to, factor out shared building blocks. These are the same patterns event-driven architectures use, and there are maybe eight or ten of them. They get you a long way, but they don't change the fundamental shape: the user is constrained to the workflows you specifically supported. The first exercise of the workshop was to build exactly this — one prompt in, one notification delivered, with `generateObject` extracting the structured data and a straight code chain calling the three backend operations in order. Most people had it working in a few minutes.

It's worth being honest about what this loses you. Five years on, this kind of pipeline still has the best price–latency profile by a wide margin. If you're running hundreds of calls per minute in production, nothing beats it. The agentic approaches that came later are more flexible and more impressive, but you pay for it. You should know which problems are still better solved this way.

## ReAct — handing control to the LLM

The next step was the arrival of tool-calling — what we now call tools. The model could ask for a function to be called, see the result, and decide what to do next. The first popular framework around this was ReAct: Thought → Action → Observation, looped until the model decides it's done.

In ReAct, the tools are still defined in your codebase. You wrap each backend function with a description and an input schema (we used Zod, but the framework converts to JSON Schema under the hood, because that's what the underlying APIs speak). The workflows — the "if someone asks to send a notification, you usually register, compile, and send, in that order" — move out of code and into the prompt. The LLM decides what to call and when. The code's job is to bound the loop, usually with something like "stop after eight steps."

In our seven dimensions:

- Tools: still defined in code.
- Workflows: in the prompt.
- Control: LLM.
- Speed: the slowest of any approach. Where the static workflow had a single LLM call extracting structured data, ReAct can easily turn that into four, five, six round-trips. Each tool call needs a "think about what to do" turn around it. LLM calls are a second or more even on smaller messages, so a five-step ReAct run is a five-second wait. That was shocking to users who'd been getting hundred-millisecond responses from the static workflow, and it's still slow today.
- Reliability: worse. ReAct gets lost. It can pick the wrong tool, jump out of the loop early, get confused halfway through.
- Context management: the LLM's problem now. It decides what gets re-read, what gets ignored.
- Dynamic: this is what you bought with everything else. With ReAct you can handle requests you didn't anticipate, because the model can decide on a sequence of tool calls that fits the request.

A small protip on tool definitions that came up while people were building this: the schema you give the model matters as much as the description. If you use Zod (or whatever your framework's schema layer is), put descriptions on every field. The JSON Schema the model sees gets those descriptions baked in, and they meaningfully change whether it picks the right tool and the right arguments. Be as restrictive as you can. If a field has to be a number between 1 and 5, encode that in the schema — then the model literally cannot get it wrong, because the validator will reject the call before it runs.

## Plan → Execute, and the todo list

ReAct's main weakness is that it gets lost. Two evolutions tried to fix that, both still recognisable in modern frameworks.

**Plan → Execute** asks the LLM to plan first: given this request, what's the sequence of tool calls you'd make? Then it executes that plan instead of re-deciding at every step. It's faster than ReAct (you skip the "think about what to do next" turn between every tool call) and a bit more reliable, because there's a written plan to follow. It's still not enforced — nothing actually stops the model from deviating from its own plan — but in practice it stays on track better. It's a step back toward the static workflow's predictability, just with the workflow generated on the fly instead of being hand-written.

**Todo lists as tools** push the same idea further. The model maintains a todo list as a tool it operates on, and you instruct it in the system prompt to re-read the todo list before every step. With a long-running task and a fifty-message history, the original plan would otherwise drop out of context entirely; the todo list survives. This works much better than nothing, and it's what the modern Claude Code system prompt does — when that feature first shipped, you could read their prompt and see what they were doing: a short instruction on how to use todos, plus an injected reminder before every tool call to check and update the list. The combination got them most of the way there, but even that wasn't a full guarantee — the model can still forget to update the todo, or pick a step out of order. Reliability has improved as todo usage has shown up in training data, but it's still not the same as enforcement.

In dimension terms, both of these inherit ReAct's profile and trade a little speed for a little reliability. They don't change the fundamental property: the LLM is still in charge, and nothing in the loop forces it to stay on plan.

## MCP — making tools portable

So far the tools have always lived inside the agent's codebase. If you wanted the same tool in another agent you copy-pasted, or shoved it in an npm package, or implemented it twice in two languages. Frameworks like LangChain offered some reusable tools — web search, that kind of thing — but it was always painful. The Model Context Protocol fixes the wiring.

The simple way to describe MCP is that it's a standard wire format for exposing tools to LLMs over a network. There's a server that publishes its tools, resources, and prompts (over stdio or Streamable HTTP). There's a client — the agent — that discovers them at runtime and calls them as if they were local. The decoupling is the whole point: whoever owns the tool doesn't have to be the same party running the agent. You can swap backends, share tools across teams, run tools in another process or language or machine. The agent-side loop doesn't change at all — it's still ReAct under the hood. Only the wiring is different.

What I want to emphasise — and it's worth saying loudly because I think MCP gets over-mystified — is that an MCP server is not a complex piece of infrastructure. It's a thin layer in front of an existing backend. You don't have to rewrite your backend to support MCP; you put a layer in front of it. We did exactly this in the workshop. The existing REST backend ships with an OpenAPI spec, and FastMCP can ingest that spec and stand up an MCP server that exposes one tool per OpenAPI operation, with all the schemas and descriptions derived from the spec at runtime. Zero codegen, a few lines of Python:

```python
from fastmcp import FastMCP
import httpx, yaml
from pathlib import Path

spec = yaml.safe_load(Path("specs/rest/openapi.yaml").read_text())
client = httpx.AsyncClient(base_url="http://localhost:3010")
mcp = FastMCP.from_openapi(openapi_spec=spec, client=client, name="backend")
mcp.run(transport="http", host="127.0.0.1", port=3020)
```

You can test the server with the MCP Inspector before any agent is involved — point it at the URL, hit connect, list tools, see all the operations. This is the right mental model: at SAP some of the work happening now is exactly this pattern at a larger scale — they build an MCP gateway that sits in front of existing backends and takes care of the protocol layer for you. You hand it an OpenAPI spec; the gateway takes care of the rest. So when you hear MCP, don't think "rewrite the backend." Think "thin layer in front of the backend."

On the agent side, swapping a typed-tools ReAct agent over to MCP is a small change. You connect with `createMCPClient`, call `client.tools()` to get AI-SDK-shaped tool definitions back, and pass them straight to `generateText({ tools })`. No Zod, no `tool({...})` wrapper, no schemas in your code — the server owns them. Always close the client in a `finally` block.

A subtlety that already matters in production agents: at SAP the rule is that HTTP calls happen *only* through MCP. The agents we're going to be building for the engagement-agent line will be MCP-only on the wire. There's a real cost to this that's worth getting ahead of — when an MCP tool returns a chunk of data (say, the rendered HTML of a template), it lands in the agent's context, and that scales badly. A megabyte of HTML in the context window is enough to derail half the runs. We'll come back to this; it's the kind of constraint that pushes you toward skills.

## Skills — workflows packaged with their tools

MCP solved tool reuse. But it didn't address two other problems that started to bite as soon as agents had access to lots of tools.

The first is context bloat. When you bind two hundred tools to an agent — and that's a realistic number once you've connected Slack, Jira, Confluence, your own MCP servers, and so on — the full JSON Schema of every tool, with all those descriptions you so carefully wrote, lives in the model's context for the whole session. That's a lot of tokens spent on tool definitions before you've said anything.

The second is tool-selection scaling. With two hundred functions available, the model has to decide which one to use. It's the same problem a human would have: if you knew two hundred programming languages and had to pick one for a new task, which one do you pick? They overlap, the differences are subtle, and the cost of getting it wrong is real.

Skills came in to address both. A skill is a markdown file — `SKILL.md` — that lives in a directory. The directory can also contain whatever resources the skill needs: spec files, example payloads, scripts. The model gets a *catalogue* in its system prompt — one or two lines per skill saying what it does and when to use it. Only when the model decides to use a skill does it actually read the full `SKILL.md` and pull the contents of the directory into context. The full body of the skill is loaded dynamically; only the catalogue entry is paid for upfront.

Critically, a skill packages not just tools but *workflows*. MCP gave you a way to expose tools over the wire; it said nothing about how those tools combined into a useful sequence. The skill body can say: "for a campaign creation, call these five operations in this order, here's the data flow between them." You can put your own tool workflows in a skill, or you can write a skill purely about how to compose someone else's tools — you don't have to own the tools to write a skill about them.

The dimensions shift, but not as dramatically as you'd think:

- Tools: code + skill (because the skill can introduce additional ways to call tools).
- Workflows: prompt + skill (you can write the workflow inside the skill body).
- Control, context management, dynamic: similar to the other LLM-driven approaches.
- Reliability: depends entirely on the skill.

A practical detail that came up: when an agent uses a skill, it needs some way to actually act on the skill's instructions — to read the spec files the skill points at, to make HTTP calls if the skill tells it to curl an endpoint. In Claude Code, that comes for free; the agent has bash. For the custom agents we're building in the workshop, you have to give it tools explicitly. The simplest version is to give it bash, and bash takes care of both reading files and making HTTP calls. That's not what you'd want in production — production agents should get tightly scoped tools (a read-only file tool restricted to the skill directory, a single HTTP tool, that kind of thing), not bash. But for the workshop a single bash tool is enough to make the whole thing go.

There's a sharper reason to prefer skill-driven over MCP-direct for some workflows: skills let the agent *chain* operations inside a single bash command, never bringing the intermediate results into context. The example I keep coming back to: suppose you need to send a templated notification, and the templated content is a chunk of HTML. With MCP, the agent calls the template-get tool, the HTML comes back as a tool result, and the entire blob lands in the model's context. The next call has to live with that. With a skill that drives bash, the agent can write a one-liner that pipes curl through `jq` to extract just the name, or saves the HTML to disk and only reads a single line out of it. Whatever the agent doesn't want in its context, it doesn't have to put there. Claude Code is especially strong at this — chaining HTTP requests, jq filters, and file operations into bash one-liners — and it's a meaningful capability that pure-MCP agents don't naturally have. The MCP spec has proposals for similar functionality, but it's not there yet.

One trap to call out, because it came up while building the skill in the workshop: when you ask the model to *write* a skill for you, it tends to write a 200-300 line document that duplicates everything from the OpenAPI spec — every endpoint, every field, every type. That's exactly what you don't want. The whole point of putting `openapi.yaml` next to `SKILL.md` is that the spec is the single source of truth and the skill can just *point at it*. Tell the skill: "read `openapi.yaml` for endpoints and schemas, read `arazzo.yaml` for the register → compile → send sequence, calls go via curl to localhost:3010." That's enough. Don't duplicate. The model reads the specs itself. Duplication is worse than useless — when the two copies drift, the resulting contradictions confuse the model badly. I saw this concretely on another team's project: their MCP server already had structured field descriptions, and their skill duplicated all of them verbatim, with no benefit and real risk of inconsistency.

## Dynamic workflows — where things are going

Everything up to this point has the same fundamental shape: an LLM in a loop, deciding what to do next at every step. The reliability problems we've been working around are all downstream of that. The plan helps. The todo list helps. They're patches.

The new generation of agents takes a different cut. The pattern is: the agent decides, *up front*, on a workflow — a piece of code, or a structured spec — that describes what's about to happen. Then that workflow is *executed* by something deterministic. The LLM isn't in the inner loop anymore. It's back to being the thing that planned, with the execution handed off to ordinary code. We get reliability back, because the workflow runs the same way every time.

You can see this idea showing up across the industry from different angles. Meta's recent paper goes in the direction of having the agent write Python code that drives the rest of the work. Anthropic's experimental feature in Claude Code does something similar but using a templating language with JavaScript-style annotations, not raw code execution. Both are reaching for the same thing: combine the dynamic capability of an agent-generated plan with the reliability of code execution.

The dimensions for dynamic workflows look like this:

- Tools: MCP / skills / typed tools — any of them, the agent picks at workflow-generation time.
- Workflows: prompt + skill, but written out as a structured spec the engine actually enforces.
- Control: code, again. Because once the workflow is generated, it's just code being executed.
- Speed: fast. There's overhead in the planning step, but during execution there's none of the per-step LLM thinking.
- Reliability: the best. The same as the static workflow — once the workflow exists, it runs predictably.
- Context management: also code's job again, with the workflow able to fork sub-agents and route context deliberately.
- Dynamic: more — because the workflow itself is generated on demand, you're not limited to the workflows you hand-wrote.

I built one of these for myself and it's the thing that's most changed how I work. The approach I use is a framework where you describe workflows in YAML — steps, control flow, parallelism, sub-workflows, building blocks that re-encode the common patterns (fan-out then merge, tournament-style "run N variants and pick the best with a judge," human-in-the-loop steps that pause and ping me). Workers running in pre-built Docker images pick up tasks. The workflow is *durable* — if one step crashes, I can resume from there, the way you'd resume on a message-queue system. Compare that to a Claude Code agent running on my laptop where, if I close the lid, the work is gone. With a worker pool I can run the same kind of work on a server with 48 cores, scale up and down, and not have any of it tied to my local machine.

A few examples from my own setup, because abstract descriptions don't land:

- **A deep research workflow.** A real deep-research run is expensive — many search queries, fetches, adversarial verifications, critic passes, synthesis. The shape ends up being a tree: the top level fans out into search queries, each query has a sub-workflow that fetches and reads sources, and the results are merged and critiqued at the top. With the framework I'm using, sub-workflows are first-class — a step inside the main workflow can call into another workflow, and the children can run in parallel on separate workers.

- **A daily tracing report.** A scheduled workflow that pulls the previous day's trace data into a file, hands the file to an agent (the agent never has to use the search API directly — it can't mess that up), the agent produces a structured report in JSON, and my own code turns the JSON into HTML. Mixing AI and ordinary code in the same workflow is the whole point. The orchestration and decision-making can be AI; the deterministic parts stay deterministic, and you don't burn tokens on things that don't need them.

- **A 55-file migration.** A migration task across the codebase: a Python step found the files to migrate, the workflow then spawned 55 parallel agent runs (five at a time, bounded by the worker pool), each one transforming one file. Each transformation was followed by a guard step — a static shell-script check on what changed, plus a tiny haiku-model agent verifying that the migration didn't introduce unrelated edits. The whole thing ran for about three and a half hours overnight while I wasn't watching, and the result was correct. I trusted it because I'd reviewed *the workflow*, not each output — the workflow guaranteed each file was processed identically.

The first time you build one of these workflows it's painful. A participant in the workshop made the point clearly: on his first run, every node had something wrong with it — outputs in the wrong shape, missing permissions, the agent in one step couldn't see the file from a previous step. He spent a full day debugging node-by-node. After the first end-to-end success, though, the pattern was reproducible. Workflows are a thing you can package as a skill once you've got a working one, so the next workflow you build inherits the lessons of the last.

Two practical observations from running this for a while:

- **You front-load thinking and you back-load watching.** The pattern looks like what we used to do in the editor team when we did heavy planning — break the task down at the planning stage, get the architecture right, agree on the divisions of work, and then trust the team to deliver. The team in that case was humans; now it's agents. The shape is the same: front-load the planning, then let the work run. The difference is that the agent never gets tired, never needs to context-switch out, and the workflow runs through the night. If you got the plan right, you wake up to it being done. If it failed, the human-in-the-loop step pings you and you decide what to do.

- **You still need a babysitter.** Even when a workflow is running, I usually have a small agent that polls it every minute or so, looking at logs, checking for steps that errored, deciding whether something needs to be retried or whether the plan itself needs updating. Most of the time it just confirms things are fine. When something breaks, it retries the failed step from that point — because the workflow is durable — and only escalates to me if the failure is structural enough to need a plan change.

There's a meta-shift here that I think is harder to internalise than it sounds. When you're working this way, you're not building the feature anymore — you're building the thing that builds the feature. You step back one level. That's psychologically different. I notice the people who love hands-on coding finding this transition uncomfortable; a couple of people I respect technically have told me recently that they're considering stepping away from programming because "this isn't programming anymore." A participant in the workshop and I went back and forth on the same point: how much do you have to understand to operate this way? My own answer landed somewhere close to what another participant put well — the bar isn't lower, it's the same. Everything you write for the agent has to be something another human could pick up and act on. If a human couldn't execute on your instructions, the agent probably can't either. The shortcut of writing instructions that work because the agent will figure out the rest doesn't hold up — those instructions are full of unstated assumptions, and the agent has fewer of those assumptions baked in than a colleague would.

## Making the agent reachable — Agent-to-Agent

The last piece is how an agent gets *called*. A CLI is fine for hacking on your laptop but it doesn't compose — other agents, other apps, other teams shouldn't have to `npx` your script.

The Google-led Agent-to-Agent (A2A) protocol is the standard that's emerging here. It's the same shape as MCP for tools, but for whole agents. Each agent publishes a JSON document at `/.well-known/agent-card.json` describing itself — name, protocol version, URL, transport, declared skills, capabilities (does it support streaming, push notifications, auth), default input and output modes. Clients fetch the card once, learn what the agent does and how to call it, then start sending messages.

At Emarsys and at SAP this is the mandatory shape for serving agents. The engagement agents we'll be building will speak A2A on the outside and MCP on the inside. Which means the stack ends up looking like: an A2A backend exposes the agent, the A2A backend's executor calls into the agent code, the agent code uses MCP to talk to operational backends, the MCP servers wrap the actual REST backends. Three layers — A2A, agent, MCP — plus the REST backend underneath. Each layer is thin. Each layer wraps the one below with a different protocol vocabulary. That's deliberate: the layers don't add logic, they add reach.

In the workshop we built an A2A server in two files. One is mostly boilerplate — build the agent card, wire up a JSON-RPC handler, mount the endpoints. The other is the executor, which is where the actual agent work happens. A non-obvious thing that bit several people: when the executor receives a message, it has to publish a `Task` event first — `kind: "task"`, `state: "submitted"` — before any status updates. If you skip that and go straight to a status update, the result manager has nothing to attach the status to, and the framework returns "Agent execution finished without a result" without telling you what actually went wrong. After the task event, you publish a `working` status, then run the actual agent, then publish a `completed` status (or `failed`) with the final message.

To actually poke at the running server, there's an A2A Inspector — an official web UI that fetches the agent card, validates it against the spec, opens a chat window, and shows you the raw JSON-RPC frames. There's no pre-built Docker image; you build it locally:

```bash
docker build -t a2a-inspector https://github.com/a2aproject/a2a-inspector.git
docker run -d -p 8088:8080 a2a-inspector
```

A small wrinkle that tripped people up: the inspector runs in Docker, so when you point it at your locally-running A2A server, `localhost` resolves to the *container's* localhost, not yours. You have to use `host.docker.internal` instead. Once it connects, the inspector auto-fetches the agent card, runs its spec checks, opens a chat, and you can send "Send a welcome message to Viktor Somodi" and watch the JSON-RPC frames stream past in the debug console.

The UI is, frankly, terrible. There's a running joke through the workshop about how every open-source debugging UI looks like it was thrown together to satisfy a demo. But it works, the protocol checks are useful, and you can drive it.

## What this means for what we're building

Every step in this arc kept the actual backend the same. We didn't rewrite it once. What changed each time was the *wrapping* — code wrapping, ReAct wrapping, MCP wrapping, skill wrapping, A2A wrapping. The layers are thin and largely convertible. Don't fall in love with any of them. Understand the problem each one solves — tool reuse, workflow reliability, context bloat, agent discoverability — and recognise that next month a new framework will solve the same problems with slightly different names and slightly different ergonomics. If you can step far enough back to see the layers, you can re-map any new thing onto the same picture.

For the engagement agents we'll be building, the concrete shape is set. They'll be A2A on the outside. They'll be MCP-only for any HTTP. They'll use skills as the primary way to package workflows. And they'll deploy as Docker images — likely outside Emarsys, in the SAP infrastructure — which sounds great when you hear "you only need to write a skill," and slightly less great when you remember that operating a service you didn't write, on infrastructure you don't own, is its own learning curve.

The dynamic-workflow direction isn't in the production stack yet, but I'd give it six months. Skills lagged the public release by three to six months before they made it into our environment, and dynamic workflows are following the same curve. They're not yet a Twitter-trend topic, but they're the missing piece for tasks that have been growing past what a single in-loop agent can handle reliably. A multi-day task is plausible now in a way it wasn't a year ago, and the way you make it plausible is to plan it out as a structured workflow and let a durable executor run it. We've been experimenting with that this week — multi-hour, sometimes multi-shift runs, completing successfully because the structure holds even when individual steps wobble.

What I'd encourage is to start playing with this now, even if it doesn't make it into production agents this quarter. The first workflow you build will be brutal — the participant in the workshop who lost a day debugging node-by-node is the norm, not the exception. The second one is much easier. By the time you've built a handful and packaged the lessons into a skill, you're operating one level above the work, and the productivity multiplier is large enough that it's hard to go back.
