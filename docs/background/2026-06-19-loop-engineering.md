# Loop Engineering

> *This article is based on a workshop held on June 19, 2026 and was generated by AI from the slides and transcription.*

Loop engineering is one of those terms that has accumulated so many adjacent meanings over the past months that it is worth taking apart carefully. People hear it and think of three different things at once: the Ralph loop, the `/loop` cron command, and the broader idea of building self-improving AI systems. They all live in the same neighbourhood, but they are not the same thing, and the most interesting one — the one I think is actually moving us toward something like AGI for software engineering — is also the one people talk about least.

I want to walk through how we got here, what the current state of the art looks like, and where I think this is heading. This is not a new feature for an editor, it is a higher-level shift: we are not focused on building features inside the IDE anymore. We are building the systems that build the features for us.

## The Ralph loop, and why it was a starting point

About a year and a half ago, a pattern called the Ralph loop got a lot of attention. The thing itself is almost embarrassingly simple — it is a `while` loop that pulls work off a queue and hands it to an agent:

```bash
while true; do
  cat PROMPT.md | agent
done
```

And `PROMPT.md` is something like "Get the highest priority ticket and resolve it." That is it. The whole idea behind the initial framing was that you have a subscription to an agent, you want to keep that subscription saturated even while you sleep, and you have an effectively infinite queue (Jira tickets, GitHub issues, a backlog) — so you let the loop chew through it forever. When the queue empties, the agent sleeps for a minute or five and tries again. Maximise utilisation.

This is, of course, not actually how you would design it from first principles. From first principles you would let a ticket arrive and immediately dispatch an agent — agents parallelise in a way humans cannot. A human works eight hours and is hard to scale; an agent can take a hundred tickets and start a hundred parallel runs. The reason the Ralph loop looks like a sequential while-loop is that it was optimised around the limits of a single subscription, not around the problem.

But it caught on, and then it evolved. The next step was to put a condition on the loop instead of `true`:

```bash
while ! npx eslint .; do
  agent "Run `npx eslint .` and fix the errors."
done
```

Now you are not pulling tickets — you are saying "I want the ESLint errors fixed, here is what done looks like, keep going until done." You are also telling the agent to fix exactly one rule per iteration; the loop handles the rest. After that, people generalised the same shape to feature development: you have a markdown document with a task list and checkboxes, the agent is told to take one item at a time, mark it done, commit, and the outer loop keeps spinning until the list is empty.

What is important to notice across all three variants is that this is an *outer* loop. The loop logic lives outside the agent. We are looping the agent; the agent is not looping itself. That distinction matters more than it first seems, because it buys you a kind of context management almost for free. Each iteration of the outer loop spawns a brand new agent. The new agent does not inherit the previous one's tool calls, its accumulated state, its scratch. It gets a clean prompt that says "here is the work, do it." Context is passed between iterations through files — a task list, a journal markdown file where the agent records what it tried and what hurt — and that is enough. You also get traceability: each step typically gets its own commit, the whole loop runs in a sandbox, and you can walk the history backwards.

The danger is that an outer loop you control is also an outer loop you can let run forever. The agent says "I cannot do this, I do not have Docker access," the loop sees the task still on the list and throws it back at the agent. The agent says no again. You burn money in a circle. So this style of loop needs an emergency-exit mechanism — the agent has to be able to say "the environment did not change between my last two attempts, this is not going to work" and have the loop honour that. The other thing that has become standard is a budget: a token cap or an iteration cap or a dollar cap, baked into the same emergency-exit logic. When you hit it, the loop quits and asks someone to look.

A participant of the workshop asked whether this loop was actually interactive — does the agent ask questions while it runs? The answer is mostly no. These loops are built to be told a goal and to grind. There is an emergency exit, and there is a *human in the loop* counterpart where the agent can ping you when it genuinely needs help — over Slack, with a link back to the project, with a specific question. But the ping is the last resort, not the first move. The whole point is that you minimise how often you have to step in. You build the system so that intervention is exceptional.

## The poor man's Ralph loop: `/goal`

After the Ralph loop became well-known, someone — probably looking at it and thinking "this is too much plumbing for a vibe-coder" — built a simplified, in-agent version. This is the `/goal` pattern that shipped in Claude Code and similar tools. The shape is the same: you tell the agent "the goal is X, iterate until X is achieved." But the iteration happens *inside* the agent now, not outside it.

The way Claude Code does this, as I understand it, is with a smaller cheaper model — Haiku — subscribed to a hook (the end-of-turn hook, I think). After each turn, the Haiku grader looks at what just happened, compares it against the stated goal, and decides whether the goal has been satisfied. If not, it bounces the work back into the agent for another round.

This is a meaningfully different thing from the outer loop, and the differences are not subtle.

First, in the outer loop your verification was hard — you literally ran ESLint, you literally checked whether the task list was empty. In `/goal`, the verifier is a Haiku model reading the session transcript. It does not re-run ESLint. So if the agent solves the lint problem by editing the ESLint config to disable the offending rule, the verifier might catch it. Or it might not. The fidelity of the check is much lower.

Second, context management is back inside the agent. The loop is internal now, so the agent owns its own context for the whole run. If you built the agent well — with sub-agent dispatch, good context hygiene, periodic summarisation — this can be fine. If you did not, the agent will eventually choke on its own history and the work quality collapses.

So `/goal` is the poor man's Ralph loop. For simple things, it is fine — you just say "achieve this" and it works. For anything in production, for anything long or hard or expensive to verify, I still want the outer loop, because I want the control. And the two compose: you can have an outer loop that hands tasks to an agent invoked with a `/goal`, where each individual task is achieved by the inner goal-pursuit and the outer loop coordinates across tasks. That is a perfectly valid shape, and sometimes the right one.

One other thing worth noting about `/goal`: it has been so widely demonstrated that the underlying behaviour is now baked into the training data. If you watch the thinking traces of recent models, even when you do not invoke `/goal`, the model is asking itself "what did the user actually want, am I there yet?" The pattern has been internalised. You do not need to use the command anymore to get most of the benefit — the models default to it.

## `/loop` is not loop engineering

There is a separate command called `/loop` and it has confused everyone. `/loop` is, essentially, a cron. You give it a schedule and a prompt and it fires the prompt on that schedule. That is it.

```text
loop = cron
```

This is not what loop engineering means. Loop engineering is not "run something at 2pm every day." If you need a cron, write a cron. I have literally never used `/loop` and I do not plan to. The naming is unfortunate because it sits next to a much more interesting concept and steals attention from it. A participant of the workshop suggested it should have been called `/schedule`, which is right.

## What loop engineering actually became

If you listen to people talking about loop engineering today, they are almost always talking about something broader than any single loop. Harness engineering — the topic that had its moment six months ago, where the focus was on building the scaffolding *around* the agent (planning, tool definitions, sub-agent profiles, context windows, all of it) — has been quietly absorbed into loop engineering. The two are now one conversation. Because if your goal is for the agent to work alone, with as little of your time involved as possible, then the loop is the outer shape but the harness is everything inside it. You cannot have one without the other.

And once the harness and the loop are in the same conversation, a third thing becomes part of it too: **self-improvement.**

## Self-improvement: the part nobody talks about yet

The self-improvement loop has two flavours, and I am running both.

The first is **scheduled improvement**, which in my case is literally a daily cron:

```text
/loop every day at 09:00
Digest yesterday's usage of the {XYZ} skill, identify recurring failures,
and propose and implement one concrete improvement.
```

Every morning a workflow runs over the previous day's traces. It looks for patterns — sequences of tool calls that should have been a single call, repetitive failing patterns, things I am habitually doing in ten steps that should be a skill. Initially it just proposes the change. Once you trust it on a particular kind of finding, you let it apply the change automatically; next morning it runs again, looks at how the change performed, and tweaks. This is genuinely a loop: it does something, checks the result, does something, checks the result.

The second is **per-session improvement**. After a particular workflow run — say, I just developed an application and it took six tries to get the Docker setup right — I have a routine that immediately analyses that specific session. Not "what happened yesterday in general" but "in this concrete piece of work, where did the agent stumble, what would have helped." I have not automated the action side of this yet. The analysis runs, it gives me findings, and I decide what to do with them. I do not yet trust it to apply changes automatically off a single session, because I think the failure mode is catastrophic — but in a quarter from now, probably I will.

Both of these flavours, scheduled and per-session, are loops in the loop-engineering sense. And the moment you let them apply changes autonomously, the system starts modifying itself.

This is where it gets very easy to make things worse. The pattern people run into — and a participant of the workshop named it correctly — is **overgrowth**. You set up self-improvement, it works, you set up another one, that works too, you set up twenty of them. Now every morning thirty new skills get generated, most of them mediocre, half of them never used, and a month later when you actually want to use one of them it turns out the underlying library has had two breaking versions and your skill is worse than nothing. You spent money to generate noise.

So self-improvement only works if there is a counter-force pulling the other way: something that prunes, deprecates, kills skills that have not earned their keep, merges duplicates, notices that you have three things that all do roughly the same thing badly. The participant who raised this called it a pulsing system — overgrows, gets cut back, overgrows, gets cut back, always rebalancing against whatever you are optimising for (cost, speed, quality). I think that framing is exactly right. The equilibrium is dynamic, never static, and the rebalancing has to be part of the engineering.

The practical advice I keep giving myself here is: do not let the agent build a self-improvement system that you yourself have not done by hand a couple of times. Review the traces for a week. Identify one pattern. Build the skill manually. Use it. Notice what is wrong with it. Iterate it manually. *Now* you understand the shape of the problem well enough to delegate it. People who skip this step end up with a hundred bad skills and no idea why anything is happening.

## Why this is all converging on cost

A few years ago we did not talk about cost much. Tokens were cheap, subscriptions were cheap, the models were giveaway-priced because the providers needed the training data. That window is closing. The models are capable enough that the providers can charge real money, and they will. Self-improvement loops, evaluation runs, multi-modal pipelines that run the same task five times through five different models and pick the best — all of these stack up. It is dangerously easy to burn a four-figure number in an evening if you have not built spending guardrails.

So a real loop-engineering setup now includes budget enforcement at the loop level, model selection per step (do not run Opus when Haiku is good enough), and a meta-loop that monitors how much you are spending per unit of value produced. A participant of the workshop pointed out that this is essentially a measurement problem on top of an optimisation problem — what do you even measure? Cost per fix? Time per feature? Some weighted combination that changes when the priority changes? — and they are right that this is a whole class of work that did not exist before.

There is real research on this. Papers from years ago, from the classical machine learning era, already proved that you do better with an ensemble of ten cheap models than one expensive one. The same logic applies now: five Sonnet runs combined intelligently can beat one Opus run. But you have to do the engineering — the merging, the voting, the deduplication, the per-step model routing. That is its own discipline. The trade-off is wall-clock time, but the cost difference is often dramatic enough that it pays for itself.

## The ROAD to AGI

I want to step back from the loop engineering itself for a moment, because something has been bothering me about how people talk about where this is all going.

There is a recurring pattern. Someone notices a capability the current models do not have. They invent a prompting technique to fake it. The technique works well enough that papers and blog posts appear. Frameworks pick the technique up and bake it in. Once it is in a framework, the providers can measure who is using it — they have the labels. So they start benchmarking it; whoever's next model is good at "chain of thought" wins on the benchmark, so they train the next model harder on chain-of-thought-shaped data, until the model just does chain-of-thought natively without needing to be asked. The technique disappears from the user-facing prompt and reappears as a native capability of the model. Now you do not write "think step by step." The model thinks step by step because that is how it was trained.

This has happened, in order, with:

- Few-shot examples
- Chain of thought
- Tool use (which used to require you to specify the JSON schema by hand, and there was a whole debate about whether to specify tools in TypeScript or JSON Schema or XML)
- Todo lists and planning
- Goal pursuit — which is the most recent one; I noticed it really starting to land around the 4.6 / 4.8 Opus generation. The model just does it now.
- Self-improvement (this is the asterisk in my list — it has not yet landed natively, but the same arc is visible)

Each of these stacks on the previous ones. Chain of thought needed to work before goal pursuit could work; goal pursuit needs to work before serious agents can work; serious agents need to work before self-improvement can land natively. The whole thing is a tower, and each new layer multiplies the surface area of what is possible. If you put a new capability in the base of the tower, the breadth of what the top can do expands superlinearly.

This is why I do not believe "you will all be prompt engineers" or "you will all be goal-definition engineers" as the long-term picture. The skill that is hot right now is exactly the skill that will be absorbed into the model next. A couple of months ago there was a model demonstrated at the Anthropic conference whose training was specifically on going from free-text user input to well-defined goals with verifiable success criteria — and honestly, that model already defines better goals than most people I know. By the time you finish learning how to be a great evaluation expert, the model will probably already be a better one. That is not a reason not to learn it — I have learned it, I am glad I did, it pays off in the meantime — but I would not bet a ten-year career on any particular layer of this tower.

What I would bet on is *understanding the whole tower*. The people who are doing best with these tools right now are not the ones who memorised the latest prompting trick. They are the ones who were here for the last five tricks and saw each one get absorbed, so they have an intuition for what is fragile, what is going to land natively, where the rough edges are this month. That intuition does not become obsolete the way a specific technique does.

## Where this lands

There is a perception arc that I think is worth naming explicitly. When ChatGPT first appeared, a handful of people — myself included — looked at it and said: this is finished, in five years we will not be writing code by hand. At that point it was barely autocomplete and most experienced developers laughed at the claim. Then the next layer landed and a few more people came around. Then the next. Each layer pushes the perception line through a different group: ChatGPT first crossed the threshold for the general public, much later it crossed for senior developers, more recently it is crossing for the specific kind of developer who said "it will never replace the for-loop in my head." The hype headlines keep moving the goalposts because the threshold for "this is real" is different for every person; the hype machine just hits everyone's threshold at a different time.

I want to be precise about my own current position. We are not at AGI for general-purpose anything. But for software development — for the specific question "can a system take a goal and ship working code without me babysitting it" — I think we are inside one or two model generations of the answer being yes. Not as a single dramatic event. As one more small step. The system that can run a self-improving loop over its own development sessions and ship a working production service is not a category leap away. It is the next quarter or two.

I had a concrete experience this week that made me sure of this. I was preparing a workshop and built a workflow that builds workflows — a meta-framework. To test it, I asked it to use itself to rebuild itself in a different framework. Once that worked, I asked it to use the rebuild to rebuild itself in a third framework. Overnight, with no supervision, it did the whole chain. It cost maybe 250 euros. At the end I had a working implementation in three different framework targets, plus a wiki-style mental model — generated alongside the code — that explained where the central design decisions in each implementation diverged from the others.

If I had done that work by hand, it would have been three days per framework plus another two days to write up the comparison. Call it eleven days, call it whatever my eleven-day rate is in euros. The 250 was not even close. And I do not know any of those three frameworks well enough to have written the comparison myself, so the output is not just cheaper — it is also something I could not have produced. That is the line. When the system can do a piece of work that I could not have produced, it is past me on that piece of work. I am not yet ready to call this AGI in general. But on that specific task that morning, it crossed the line.

## What this means if you are a developer right now

The most concrete advice I can give — and I gave it again in the workshop — is to spend as much of your time as you can afford on learning this. Not on shipping the next feature using the old tools faster, but on shipping the next feature using these tools at all, even when it takes twice as long the first time. The doubling is the cost of learning. After three or four iterations the same task gets faster than it would have been the old way, and you have absorbed the pattern.

A participant of the workshop described this exact arc from their own week: the first workflow took an entire day of babysitting just to get it to run end-to-end. The second workflow built on the same harness took a few hours. The third will probably take half an hour. The learning curve is steep but its slope is in your favour, and the only way to climb it is to do it.

I have also noticed that the most productive use of these tools, for me, is not "ship the same kind of feature I would have shipped, faster." It is the small bits of self-tooling that compound. When I find myself doing the same ten-step thing for the third time, I stop, write a quick skill or a tool that wraps it, and move on. The skill might be ugly, it might be ten minutes of vibe-coded glue, but it saves me time forever after. Over a month these compound into a personal harness that nobody else has, and that is the actual leverage. A participant of the workshop framed this as "watch yourself for what you keep wasting time on, then carve ten minutes to fix it" — and that is right, that is the loop you should be running on yourself.

There is a counter-argument I have heard from very strong developers I respect, which is roughly: I do not enjoy this. I do not want to babysit an AI; I want to write code. The job that I will have in a year is not a job I want to have. I take this seriously. Two of the best developers I know have told me, plainly, that they are considering leaving the field because this version of the work does not appeal to them. It is a real cost and it is one of the more painful conversations to have. The honest answer I can give is that the developers who *do* lean into this can move from babysitting toward something more like loop engineering itself — building the systems that obviate the babysitting, rather than doing the babysitting forever — and that work is, I think, more interesting than the work it replaces. But the transition is not free and the loss is not nothing.

## A closing observation about everyone else's job too

One of the things I keep noticing is that this conversation generalises beyond developers. Everything that lives on a computer and is mostly logical steps over information — accounting, parts of legal work, parts of analyst work — is exposed to the same pattern. There will always be a lawyer who talks to the judge; there may not always be a lawyer who drafts the contract. There will always be an accountant who signs off on the books; there may not always be a junior who builds the spreadsheet. And what happens to the social arrangements that depend on those jobs is a much bigger question than I can answer here.

We are not done with loop engineering. The papers I am reading right now are heading further — evolutionary frameworks, genetic algorithms wrapped around agent populations, search procedures over the harness itself. Self-improvement is the layer I have actually deployed; the next layer will be agents that design *other* agents and pick the best of them under some fitness function. I do not know yet what the layer after that looks like. But the rate at which these layers are landing is faster than I have ever seen anything land, and each one multiplies the previous ones rather than adding to them, so the curve is genuinely steep. That is not hype. That is what I see in the work this week.
