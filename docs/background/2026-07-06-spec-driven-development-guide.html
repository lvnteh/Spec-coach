Spec-Driven Development

A workflow where the specification comes first, the implementation comes last, and guards verify everything in between.

01

The idea

In most development workflows, the AI is handed a goal and builds toward it. The problem is that "build a greeting API" has infinite valid interpretations. The AI picks one, ships it, and you find out what it built when you read the code.

Spec-driven development reverses the order. You define exactly what the system must do — in structured, machine-readable form — before any code is written. The AI's job is not to interpret your intent. Its job is to make the guards go green.

The framework has four pillars:

Specify

Write what the system must do

In structured form the AI and humans can both read. OpenAPI, Gherkin, Arazzo.

Validate

Make the spec executable

Every spec must be checkable against the running system. If it can't be verified, it isn't a spec — it's a comment.

Restrict

Bound what the AI can touch

One file. One job. The AI implements business logic and nothing else. Everything else is generated or framework-owned.

Building Blocks

Keep boxes small and isolated

Small, independently specifiable units. The AI can only break a small slice at a time. The job is connecting blocks, not designing them.

02

One file rules everything

openapi.yaml is not documentation. It is the contract. Every other artifact in the project derives from it. If a shape isn't in the spec, it doesn't exist as far as the framework is concerned.

openapi.yaml
generates / constrains
TypeScript types Zod schemas Route definitions Contract tests Workflow tests API docs

The spec defines the shape of requests and responses. Kubb reads it and emits TypeScript types and Zod schemas. The route file imports those schemas — it doesn't define its own. When the spec changes, you run npm run generate and everything downstream updates.

specs/openapi.yaml spec 
paths:
  /hello:
    get:
      operationId: getHello
      parameters:
        - name: date
          in: query
          required: false
          schema:
            type: string
            pattern: '^\d{4}-\d{2}-\d{2}$'
      responses:
        "200":
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/GreetingResponse"
        "400":
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/ErrorResponse"

components:
  schemas:
    GreetingResponse:
      type: object
      required: [greeting, holiday, daysUntil]
      additionalProperties: false
      properties:
        greeting:  { type: string }
        holiday:   { type: string, nullable: true }
        daysUntil: { type: integer, nullable: true }
Rule Anything you want to be true about the system must live in a structured artifact you can validate against — not in the prompt you give the AI. Every time you find yourself adjusting the prompt to fix a specific behaviour, ask: is there a spec I should be writing instead?
03

The behaviour spec

The spec defines the shape. It says nothing about what the system should do. For that you need a second layer: a concrete list of inputs and the exact outputs each one should produce.

This is the PM's document. It answers one question: for this input, what is the exact output? It is written before implementation. It is what "done" means.

The format is Gherkin — Given / When / Then. Each line maps directly to something you already have:

specs/features/hello-greeting.feature behaviour spec 
Feature: Seasonal greeting

  # Scenario Outline = one template, many data rows
  Scenario Outline: Today is a holiday
    When  I call GET hello with date "<date>"
    Then  the status is 200
    And   the greeting is "<greeting>"
    And   the holiday is "<holiday>"
    And   daysUntil is 0

    Examples:
      | date       | greeting         | holiday    |
      | 2026-01-01 | Happy New Year!  | New Year's Day |
      | 2026-12-25 | Merry Christmas! | Christmas  |
      ...24 holidays total

  Scenario Outline: Date param — invalid input returns 400
    When  I call GET hello with date param "<date>"
    Then  the status is 400
    And   the error is "<error>"

    Examples:
      | date       | error                    |
      | banana     | Nem valid datum, BLEGH   |
      | 2025-12-25 | A Dátum csak idei lehet. |

Gherkin is BDD — Behaviour-Driven Development. It is TDD applied at the feature layer, with a syntax non-engineers can read. The PM writes the scenarios. The engineer writes the step definitions that wire each line to real code. That division is the point.

Scenario Outline vs. individual Scenario: Use an Outline with an Examples table when the same behaviour repeats across many data points. Use an individual Scenario when the case is structurally distinct — different assertions, different step shapes.

What goes in the feature file Business rules — greeting logic, error messages, validation behaviour. Not protocol conformance (Schemathesis handles that). Not CSS or layout (not behaviour at all).
04

The guard stack

The spec is only useful if it runs. Each guard verifies a different layer of the system. Together they tell you whether the implementation matches the spec — and which layer broke if it doesn't.

guard:typecheck
TypeScript. Catches missing types, wrong return shapes, handler mismatches with route definitions. Runs in milliseconds. Always first.
guard:features
Playwright BDD. Runs every Gherkin scenario against the real running server over HTTP. Catches business rule violations and HTTP-layer bugs. PM-authored scenarios become automated acceptance tests.
guard:contract
Schemathesis. Reads the OpenAPI spec and auto-generates inputs — arrays where strings are expected, empty values, very long strings, every edge case humans wouldn't think of. Catches protocol conformance issues.
guard:workflow
Arazzo / Redocly Respect. Declares multi-step workflows and asserts their end-to-end status codes. Catches cases where a sequence of calls breaks even if each individual call is valid.

These are complementary, not redundant. Each one catches a class of failure the others miss.

The blind spot that taught the most The feature guard calls the real HTTP server. An earlier version called pure functions directly and missed the HTTP layer entirely. When a Zod schema on the route silently intercepted invalid input — returning a framework error instead of the custom error message — the guard showed green. The browser showed broken. The lesson: every guard that bypasses the HTTP layer has a blind spot.
05

The testing pyramid

Many tests at the bottom where rules live. Few at the top where display lives. This isn't about effort — it's about where complexity actually lives in the system.

UI behaviour
4 scenarios
Workflow / status
4 workflows
Contract / protocol
auto-generated
Business logic
33 scenarios

The UI has four scenarios — not because the UI wasn't thought about carefully, but because the UI has no rules. It calls the API and displays the result. The only questions worth asserting: does it call the API, does it show the result, does it handle errors. Everything else is implementation detail.

The business logic has 33 scenarios — because the 10-day window, the tie-breaking rule, the boundary conditions, the year validation, and the error messages are all rules that can each independently be wrong.

06

What each file is, and who owns it

One of the clearest benefits of spec-driven development is that file ownership becomes explicit. Every file in the project has a clear answer to: who wrote this, can I edit it, will it be regenerated?

File Owned by Purpose
specs/openapi.yaml PM + Engineer Single source of truth for API shape
specs/features/*.feature PM Behaviour scenarios — written before implementation
specs/workflows/*.arazzo.yaml Engineer End-to-end workflow status checks
src/generated/ Kubb (auto) Types + Zod schemas — never edit manually
src/routes.ts Framework Route wiring derived from spec — never edit manually
src/implementation/handlers.ts AI / Engineer Business logic — the only file the AI touches
src/steps/*.ts Engineer Step definitions wiring Gherkin lines to real calls
docs/decisions/ADR-*.md PM + Engineer Why decisions were made
07

Decisions that disappear without ADRs

The system has no memory. Files capture the current state. They don't capture the why. Every significant PM decision that changes the shape of the spec will leave traces in the files — but no explanation of why those traces are there.

An Architecture Decision Record (ADR) is a short document committed alongside the code change that triggered it. It captures four things:

ADR-001 Accepted

Feature scenarios use 2026 dates only

Context

The ?date param validates 2026 only. When cases.json was replaced with Playwright BDD, every date in the feature file began going through the full HTTP pipeline including year validation. 2025 dates now return 400.

Decision

Feature scenarios use 2026 dates exclusively. 2025 holiday data remains in the handler table but is not exercised at the HTTP layer.

Options rejected

Expand the year rule to allow any supported year — rejected because it changes product behaviour and the scope was too broad for the migration.

Consequences

If the year rule ever expands, the feature file needs new rows. The 2025 data is untested at the HTTP layer.

The Consequences section is the most valuable part over time. It is how future-you answers "why is it like this?" without archaeology through git blame.

08

The correct sequence

Spec-driven development has a strict order. Breaking it doesn't produce an error — it just produces a project where the spec describes what was built rather than what should have been built.

1. Spec
Write openapi.yaml. Define every endpoint, every parameter, every response shape, every error. This is the contract. Nothing gets built until this exists.
2. Behaviour
Write the Gherkin scenarios. For each input, declare the exact output. This is what "done" means. Every ambiguous case must be resolved here, before implementation begins.
3. Generate
Run Kubb. TypeScript types and Zod schemas are emitted from the spec. The route file is wired. The AI has a strongly-typed contract to implement against.
4. Implement
The AI implements one file: the handlers. It iterates until every guard is green. If a guard fails, the implementation is wrong — not the spec.
5. Record
Any decision that changed the shape of the spec during this cycle gets an ADR. Committed alongside the code. The why is preserved.
The spec-after trap Writing scenarios after building the UI feels like spec-driven development. It isn't. Spec-after describes what was built. Spec-first decides what to build. The difference surfaces on the hard cases — edge conditions, error states, ambiguous inputs — which are resolved before implementation in spec-first, and skipped entirely in spec-after.
