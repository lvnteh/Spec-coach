- spec-driven development
	- idea: don't build the app, build a framework in which the AI builds the app
	- framework is opinionated, limits AI's blast radius
	- most code becomes derived from a spec — little is left to write by hand
	- not invented here; similar tools exist
		- [GitHub Spec Kit](https://github.com/github/spec-kit)
		- Matt Pocock's `grill-me` skills repo
- four pillars
	- specify: write what the system must do in a structured way both AI and humans can read
	- validate: make the spec executable — checkable against the running system
	- restrict: bound what the AI is allowed to do — only ships things that pass the checks
	- building blocks: keep the system made of small, independently specifiable, independently testable boxes
		- e.g. diff logic for change delegation specified alone with input/output JSON pairs
		- once specified, AI can iterate, even hit a perf target
		- endpoint using it then has tiny spec — just schema + module call
		- vs. "build the whole backend" → AI wires everything to everything, hard to untangle
		- smaller, isolated blocks → AI can only break a small slice
		- programmers already do this (separate packages, domains) — just lean harder
- what this is not
	- not "AI builds apps, we have no work"
	- effort shifts: into the framework — specs, generators, guards
	- not free: first app takes ~75% of the hand-written time
		- payoff: next app in same framework is dramatically cheaper
		- can maintain 5–6 apps instead of 1 — they share the scaffold
- the numbers (real example: weekend campaign service for change delegation)
	- specification (JSON Schemas, OpenAPI, Arazzo): 26.5%
	- statically generated (routes, controllers, types): 45%
	- hand-written tooling (glue for generators): 3%
	- implemented by agent (real business logic): 5%
	- proportions depend on app type — CRUD + workflow = ~5% agent code is real
	- rewriting existing apps in this style since December — boilerplate share usually large
	- simple domain agent on existing API: implementation share to write = 0%
		- OpenAPI `description` fields describe tools
		- Arazzo workflows describe chaining
		- rest is generated
		- hard part: writing those two specs well
- Spec Kit positioning
	- probably most-used tool in this space
	- five-step shape: constitution → plan → tasks → implementation → final pass
	- good: spec/codegen separation exists
	- weak: early version was markdown-driven — worst possible format for spec work
		- markdown can only be fed to a model; can't be used for anything else
	- now adds contract-like formats (e.g. OpenAPI) — right direction, but bolted on
	- author skips the task-decomposition phase
		- AI is faster at decomposing goals into tasks
		- task list useful today only to watch/pause/take over
		- with a good framework, task layer can go
	- worth using; just not structured enough — too little focus on static generators
- step 1: bootstrap
	- start: a hand-written server that works
	- `npm install`, start server, run `npm run guard:manual` in another shell
	- manual guard = small shell script that hits the live app and confirms it compiles a template
	- most primitive validation — every later step adds a stronger guard
- step 2: specify API surface, validate with Schemathesis
	- Schemathesis: Python package (pip or Docker)
		- reads OpenAPI spec, generates property-based tests, fires them at running server
		- walks every field — empty strings, long strings, negatives, zero, edge cases
		- finds things hand-written tests miss (no human prior about "interesting" inputs)
	- useful flags
		- `--url <base>` — running server
		- `--phases examples,coverage,fuzzing` — deterministic + fuzz, skip stateful
		- `--seed 42` — reproducible (drop in production-style runs)
		- `--workers auto` — sane defaults
	- trap: Schemathesis 4.x chokes on cross-file `$ref`
		- bundle/deref first with `redocly bundle -d`
	- different participants got different test counts (250, 950, 1152) — depends on phases enabled
	- type-agreement is real friction
		- `format: email` interpreted differently by Schemathesis vs Zod/Hono
		- for change delegation: had to patch both to make chain compatible
		- this integration tax is part of "building the framework" — paid once
	- side effect: schema-shape tests you already have can usually be deleted
- step 3: specify workflows with Arazzo
	- OpenAPI describes single endpoints; Arazzo describes how endpoints chain
	- YAML file with steps
		- each step calls an `operationId` from one or more OpenAPI specs
		- pass parameters in, capture outputs, next step references them
		- per-step success criteria, even retry logic (e.g. on 500 back off + retry)
	- session example: create template → capture id → call compile with id + contact → assert personalized text
	- complex example (change delegation)
		- parameterize whole workflow (e.g. takes campaign name as input)
		- declare workflow outputs
		- call from code: factory generates random params from OpenAPI, feed into workflow, capture output, assert
	- executor: Redocly Respect (ships in Redocly CLI)
		- `--server <url>` overrides spec's `servers:` for live run
		- OpenAPI spec must declare `servers:` — Respect refuses otherwise
		- pin Arazzo `1.0.1` — Respect warns on `1.0.0`, next minor had tooling break
	- second use: Arazzo as agent boundary
		- HTTP tool → agent can call anything
		- Bash tool → agent can do anything
		- `arazzo.execute` tool: input must validate against structured schema and reference your OpenAPI
		- agent is now in a much smaller box — only composes declared workflows
		- not "the" future, but an interesting way to keep agents inside the lines
	- Arazzo isn't perfect or unique
		- e.g. Google Cloud Workflows — similar YAML workflow shape, async, visualizations, server-side only
		- point: something structured both AI and humans can read
	- side-task: rewrite manual shell guard as `guard:workflow` — Arazzo file executed by Respect
		- use `wait-on` (or equivalent) to wait for server up
- step 4: generate types with Kubb
	- framework starts paying for itself here
	- Kubb: takes OpenAPI, runs plugins to emit code (clients, schemas, types, faker, custom)
	- three plugins that matter
		- `@kubb/plugin-oas` — reads/validates OpenAPI; must run first
		- `@kubb/plugin-zod` — emits Zod schemas for bodies, params, per-status responses
		- `@kubb/plugin-ts` — emits TS interfaces for every component schema
	- plugin options that bite
		- `version: "4"` — pin Zod v4; auto-detect can fall back to v3 silently
		- `typed: false` — keeps `ZodObject` brand that `@hono/zod-openapi` needs
		- `wrapOutput` — append `.strict()` to objects with `additionalProperties: false`
			- without it, Zod v4 silently strips unknown keys → contract drift, no warning
	- exercise: delete `src/zod/`, `src/types/`, `src/schemas/` entirely
		- regenerate with `kubbgenerate.config` + `npm run generate`
		- since they're in git, diff before/after to audit
	- warning: AI is willing to "cheat"
		- one participant: Claude found the reference repo in git history and copied generator config
		- another: AI told to maximize a score → AI edited the file that calculated the score
		- same instinct → SQL injection if data is from a DB
		- bound the agent; small targets, watch what it reads
	- subtler trap
		- AI swapping `additionalProperties` schemas or rewriting id to UUID can break Schemathesis
		- real Zod ↔ Schemathesis fault line around UUID vs string
		- reference repo uses `string` for ids deliberately — keeps chain simple
	- throw away duplicate code
		- same shape currently lives in: domain types, Joi schema, Mongo schema, description doc
		- 3–4 parallel descriptions, drifting whenever anyone edits one
		- with Kubb: describe shape once, derive Zod, TS, API validator, faker factories, even client
	- you don't have to pick OpenAPI as source of truth
		- Zod itself can be source, OpenAPI generated from it
		- the point: pick one source, derive everything else
- step 5: generate the server
	- `src/routes.ts`, `src/app.ts`, `src/start.ts` shouldn't be hand-written either — derive from OpenAPI
	- exercise: write the generator
		- parse YAML, walk every `paths.<path>.<method>`
		- render `createRoute({...})` per operation referencing Kubb-emitted Zod schemas
		- render `app.openapi(<op>Route, <op>Handler)` to wire up
		- one `app.all(<path>, ...)` per path for 405 fallback
		- stamp boilerplate (`defaultHook`, `notFound`, `onError`, `app.doc`)
		- three-line `start.ts` that boots `buildApp()`
	- author overcomplicated first attempt; simple combo works fine
		- YAML parser → structured form → tiny preprocess → Handlebars templates
		- hand AI the spec + "render `routes.ts` with Handlebars" → it does it
	- gotchas
		- OpenAPI uses `{id}`; Hono uses `:id` — transform path
		- `Allow:` header = documented methods + `OPTIONS` (+ `HEAD` if `GET` present)
		- run this *after* Kubb — schema names must exist when imports are emitted
	- why Hono
		- separates route definition from handler
		- route declares OpenAPI shape (request schema, per-status responses, errors)
		- handler is separate function wired in
		- route is generated, handler is implemented — different files, different layers
	- after this step: everything generated except handler bodies
		- only business logic is hand-written
- step 6: hand the glue to the AI
	- one file left: the handlers (real per-endpoint logic — look up template, substitute fields, return)
	- full guard set
		- `npm run guard:typecheck` — `tsc --noEmit`
		- `npm run guard:contract` — Schemathesis against OpenAPI
		- `npm run guard:workflow` — Respect against Arazzo
		- `npm run guard` — all of the above
	- prompt the AI: here is the spec, all of `src/` is generated, you may only modify the handlers file, here are the guards, iterate to green
	- restrict pillar in concrete form
		- run agent in Docker container with user that only has write permission on `implementation/`
		- agent literally cannot escape and edit the framework
	- AI takes ~1 minute on a task this size
		- most of it reading every file (guards, OpenAPI, Arazzo) before writing
		- actual implementation is quick
	- wipe handler file → guards red → AI brings to green → done
- specs vs. prompt-tweaking
	- participant noticed AI used incrementing counter for ids instead of UUIDs
	- asked AI to rewrite its prompt to prefer UUIDs
	- wrong instinct
		- fix isn't "tell AI to use UUIDs next time"
		- fix: put requirement in the spec
	- prompt-only fix → next contributor has no idea, fresh agent run has no idea — truth stored in wrong place
	- "in the spec" depends on what you're expressing
		- UUID id → schema constraint or example with real UUID
		- behavioral property → Arazzo test case or input/output pair
	- whenever you adjust a prompt to fix specific behavior, ask: is there a spec I should write instead?
		- almost always yes
- when the spec itself is wrong
	- team author works with has easy case: real backend with known shape, spec written *for* that shape
	- building from scratch: spec can contradict itself
		- OpenAPI says one thing, Arazzo workflow assumes another, guards refuse to align
		- AI churns trying to implement the impossible
	- workaround: AI keeps a journal as it implements
		- prompt: experimental harness; statically generate what you can; rest implement from spec
		- if you cannot solve something — spec contradicts itself, guard wrong, structural problem — stop, log, ask human
		- first run stops where expected, human fixes spec, next run gets further
	- more advanced: AI bails silently on blockers
		- every run is traced
		- separate agent reviews traces periodically, surfaces patterns
		- e.g. "this failure keeps happening; here's a tool you could add"
		- framework grows by mining its own failure modes
- regeneration vs. incremental modification
	- open question when spec changes after app is implemented
	- option 1: regenerate the whole thing
		- clean, deterministic
		- expensive on large apps; paid on every change
	- option 2: add new endpoint on top, AI extends existing implementation, guards catch regressions
		- cheaper per change
		- accumulates dead code — AI doesn't refactor unprompted, just makes guards pass
		- over many edits, drifts from what fresh generation would produce
	- probably a third mode: occasional regeneration, accumulate in between
	- honest take
		- if app meets perf, cost, correctness → generated code quality matters about as much as compiler-emitted assembly
		- not there yet — next few years, generated code must be readable
		- so: better generators + occasional regeneration vs. infinite drift
- frontends are further away
	- question: could a PM, with tooling + agent, kick off a feature by writing Gherkin?
	- backend: yes, approximately — numbers support it
	- frontend: extra complexity, no good tooling yet
		- backend spec is easy: endpoints, schemas, workflows
		- "what should this screen do" is much harder
		- Gherkin is best we have — generates test code *skeleton*, not implementation, not UI
		- agent-implemented share on frontends today: ~70% (vs. 5% on backends)
		- pillar shape same; tooling less mature
	- still: PM learning to write OpenAPI lets them contribute to backend directly
	- diff-spec format (input/output JSON pairs) is something a PM can absolutely write
	- structured-spec part is easier to pick up than code-generation part
- where this leaves UI in general
	- pushback: agentic world → UI matters less? smaller, review-shaped UIs vs. full editors?
	- partly yes
		- author's company is an integration platform — opening up so customer systems drive it
		- customers have their own UI → heavy-UI investment less central for the platform
		- less "user clicks 'create email' in our editor"
		- more "customer's system asks our agent → user reviews/annotates in *their* UI"
	- other side: more UIs overall, not fewer — because building each one is cheap
		- author builds his own tools → cheerfully generates a UI for them
		- faster than chat-only; visual feedback is better
		- more UI in the world; less of it ours to build
	- for the UIs we do own: higher-level design system pays off
		- most of an app = list pages, detail pages, same shapes over and over
		- if "list page" is a real building block with a few config knobs → AI generates from small config, not raw primitives
		- someone builds the high-level building block first → real investment
		- then every page is cheap and consistent
		- same Building Blocks pillar, applied to UI
- what to do today
	- even without the agent, this is worth doing now
	- if you have an OpenAPI spec — at SAP it's mandatory — actually use it
		- generate the client
		- generate test fixtures
		- generate request validators
		- hand to customers (e.g. BMW) with an Arazzo workflow that documents how to chain calls
		- replace dozen hand-written schema tests with Schemathesis
		- pull duplicate type definitions to a single source
	- none of this needs Claude — but becomes dramatically more valuable once Claude is in the loop
	- two takeaways
		- have a real spec
		- prefer the most structured representation you can stomach
			- markdown is the *last resort*, not first
			- once a fact is in markdown, it can only be fed to a model
			- JSON Schema, OpenAPI, Arazzo, Gherkin — anything structured beats prose
			- structure lets you generate, validate, constrain — that's the whole game
