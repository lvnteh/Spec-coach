# Giving Agents Access to Our Knowledge

> *This article is based on a workshop held on May 15, 2026 and was generated by AI from the slides and transcription.*

In a software company, knowledge lives everywhere — Jira, Confluence, Slack, email, code repositories, and in people's heads. Until recently, humans were the connective tissue: you were the one who knew which Confluence page to open, which Slack thread to scroll back to, who to ping. There was no central place where you typed a question and got an answer that crossed all of those surfaces. You were the search.

Now that we want agents to do meaningful work for us, that knowledge has to be reachable by something other than a human. An agent can't read your Jira. It wasn't in that Slack thread. It doesn't know what we decided last Tuesday. If you don't hand it that context, it hallucinates — it invents a plausible-sounding answer about how the editor works, or about how Emarsys does X, because the publicly trained model has none of our internal repositories, decisions, or domain specifics. The whole point of building a knowledge base is to give the agent a way to gather, verify, and work from material that's actually ours.

A reasonable counter-argument is: why not just train the model on our knowledge? It's been said many times that the big players would do exactly that. The math doesn't work. Fine-tuning a model on internal data costs in the millions of dollars per run. Even ignoring the cost, our knowledge doesn't change weekly — it changes hourly. A support workflow built around tickets from one minute ago can't wait for the next training pass. There's still value in occasionally retraining a model on broad, slow-moving company concepts, but for live, up-to-date documentation, training is out.

There's a happy side-effect to all of this. The classic problem with documentation has always been that nobody wants to write it, nobody wants to read it, and it drifts the moment the code changes. LLMs are good at exactly the things that made documentation painful — summarizing, structuring, connecting information, keeping things consistent. If we solve retrieval, the LLM can also help maintain the knowledge. The two reinforce each other.

What follows are four approaches to giving an agent access to our knowledge, in roughly the order they evolved. Each is still useful today; the real win comes from combining them.

---

## Embedding + RAG

This is the oldest of the four — embedding pre-dates the current LLM renaissance by a fair margin. The concept is straightforward: take some text, turn it into a vector that captures its meaning in many dimensions, and at query time turn your question into a vector too. The documents whose vectors are closest to the question's vector are the relevant ones.

Each dimension of the vector loosely captures a concept — animal-ness, has-wheels-ness, gender, royalty, and so on, except thousands of them. We don't actually know what each dimension means; the model learns them during training. We just know that two vectors with similar numbers across all those dimensions represent things with similar meaning. The classic example is that `king − man + woman ≈ queen` — the geometry actually holds because the dimensions of "royalty" and "gender" are real, learned axes in the embedding space. Distance is typically measured with cosine similarity, and the threshold for "close enough" is genuinely use-case dependent — you have to tune it for your data.

The complication is that an embedding for a whole document is useless. If you embed a thousand-page Bible and then search for "father," everything matches and nothing is useful. So documents have to be chunked. Chunking is its own hard problem: cut on character boundaries and you split paragraphs in half; cut on paragraph boundaries and the chunks get unwieldy; either way, you usually want some overlap between chunks so context isn't lost at the seam. Once you have chunks, you embed each one, store the vectors in a vector database, and at query time return the nearest chunks — and then you have another choice about whether to hand the agent just the chunk, the chunk plus its neighbours, or the whole document.

That whole pipeline used to be miserable to build. For years it meant orchestrating document conversion, chunking strategies, embedding model choices, and vector database operations. I built this for myself more than once and it was always painful. There are now hosted services that do the whole thing — Google's File Search is the first one I've used that's genuinely plug-and-play; you drop in PDFs, it processes them, and you get a chat interface and a search API back. The problem is we can't use it at the company for compliance reasons.

For the local case, [`qmd`](https://github.com/tobi/qmd) is what made me put my own RAG pipeline in the bin. It runs everything locally — the embedding model, the reranker, the query rewriting, all on your machine — and stores the index in SQLite. It chunks at ~900 tokens with 15% overlap, does query expansion (generating synonyms and related terms), reranks the top results, and has BM25 keyword search as a fallback. It exposes itself as an MCP server too. The first query is slow because it downloads a few gigabytes of model files, but after that it's quick. The install flow is a one-liner npm install, then `qmd collection add` to point it at a folder, `qmd embed` to build the index, and `qmd query` to search. I use it heavily for my own meeting notes and transcriptions — everything is indexed, and the agent can answer questions across it in seconds.

A few things worth knowing. First, embedding alone isn't actually that good on its own — semantic search is somewhere around 50–60% on a 5-point feel scale for retrieval quality. Keyword search isn't dramatically better. The trick is combining them: run a keyword search and a semantic search in parallel, merge and re-rank the results, then hand the merged set to the agent. That hybrid genuinely outperforms either alone, and it's what `qmd` does internally.

Second, the 2.0 picture has gotten more sophisticated. Query decomposition splits a complex question into smaller sub-queries — "what should I do next week" becomes "what is next week" and "what should I do" and several related variants. Query expansion adds synonyms and alternative phrasings. The retriever pulls a much wider set of candidates than the final top-5 — often 50 or 100 — and a reranker model re-scores them for precision before they go to the agent. These pieces all run as small, fast local models. Together they make embedding-based retrieval genuinely usable for many cases, though still not perfect.

Third, this approach has a ceiling. Embedding works well up to around 100,000 documents. Beyond that, everything starts matching everything and quality degrades. For most internal use cases we're nowhere near that limit, so it's fine.

Vector databases turned out to be a huge business — Pinecone took a lot of capital around this concept. Postgres has a vector extension (pgvector) that works well enough for our scale; Redis was an early player; even SQLite is fine for personal use. The point is that storing and quickly searching vectors at scale is a hard enough engineering problem that an entire category of databases now exists for it.

We worked through `qmd` hands-on during the workshop. The flow is: install it, run `qmd collection add` against a folder of documents, run `qmd embed` to vectorise them, and then query through the CLI. The CLI is genuinely AI-friendly — it takes flags for output format (JSON, markdown, files), collection filters, thresholds, and result counts. JSON output is the natural shape for an agent to consume. You can hook the MCP server up to your agent (Claude or otherwise) and have it query the collection from inside a skill.

One observation from the room: a participant pointed out that when you eventually send a result up to Claude in the cloud, that data does leave your machine. The local part stops at retrieval; the answer-generation step still goes to whatever model you're using. The fix, if you care, is to use a local model for the generation step too — but at that point you're trading quality for privacy.

Another participant asked whether queries in Hungarian would work against an English-language corpus. On paper they should, because the embedding maps concepts rather than words, and Hungarian-English concept distance is small for most concrete things. It works less well as languages get conceptually further apart — for languages without distinct number words, for example, the embedding model can't represent quantities the same way. In practice English embeddings are noticeably the best quality; everything else trails off.

## Agentic Search

The agentic search approach inverts the question. Instead of pre-processing documents into a searchable index, you give the agent tools — file listing, grep, file reading, whatever you have — and let it figure out what to look at on its own. Iterate until it has enough context.

This is what you're already using whenever you ask Claude something about a codebase. There's no embedding step, no chunking, no vector database. The agent just walks the filesystem.

The shape this took at the high end is interesting. Some people built a whole "knowledge layer" sitting in front of the raw tools, with its own LLM inside it deciding how to use them. The user-facing agent calls into the knowledge layer as a single tool, and inside the knowledge layer another agent navigates the actual storage. There are emerging standards for how to build these layers efficiently. That's the most agentic-search-ish version of agentic search: turtles all the way down.

The trade-off versus RAG is sharp. Agentic search is more flexible — practically anything you can give the agent a tool for becomes searchable. It scales effectively without bound, because the agent decides how deep to go. It's relatively simple to set up because you just plug in tools rather than building a pipeline.

But it's slow. A web app that needs to respond in two or three seconds can't wait for an agent to list, grep, and read its way through a filesystem. RAG returns in milliseconds. Agentic search takes seconds, and for hard questions it takes minutes or hours — the agent can keep working as long as you let it, building indexes for itself, taking notes, going deep. That's also its strength: given enough time, it will find things RAG would miss.

The mature view that emerged from this period was that agentic search and RAG aren't competitors — they're layers. RAG becomes one of the tools the agent has. The agent tries RAG first, sees whether the results look complete, and if not falls back to grepping the filesystem. The combination is better than either alone.

## Karpathy Wiki

The third approach is more recent — six months or so old — and Karpathy popularised it, though he wasn't the first to attempt it. The idea: instead of vectorising the source documents (RAG) or letting the agent crawl them at query time (agentic search), pre-process them into a wiki.

A wiki is a collection of interlinked pages, one topic per page, with explicit cross-links. Think Wikipedia but for your own knowledge. The structure is exactly the kind of thing humans use to navigate large bodies of knowledge — you open the encyclopaedia, find an entry, follow a link, follow another, and ten hops in you're somewhere useful. There's an index page that lists everything and a one-line summary of each entry; that's the agent's map.

At query time, an agent traverses the wiki the way you would. It hits the index, picks a page, reads it, follows links to related pages. The pre-built structure gives the agent a faster path through the knowledge than a blind grep would, and because the wiki itself is plain markdown, you can read it.

That last point is the real differentiator. RAG and agentic search both produce knowledge representations that are useful to the agent but not to you — vectors are opaque, and an agent's grep trail isn't a thing you can audit. The wiki is yours too. You can sit down, browse it, fix it, and watch how the agent's understanding evolves. Right now, while we're still in the phase of transferring tacit knowledge into machine-readable form, that's valuable. Ten years from now, when the agent is genuinely better at this than we are and we just want to ask questions, we may not need the human-readable layer anymore. We need it now.

The wiki has four operations:

- **Ingest** — take a new piece of source material (a meeting note, a PDF, a scraped help portal page, a code repository) and merge what it teaches you into the wiki. The agent reads the source, decides which existing pages to update, which new pages to create, and how to link them.
- **Query** — ask a question against the wiki. The agent uses the index and the page links to navigate.
- **Lint** — audit the wiki's health. Are the links valid? Are there orphan pages? Are there contradictions between pages? This is the closest thing to a maintenance script and it's important because the agent, while editing, sometimes forgets to clean up — leaves dangling links, drops references. Lint catches that.
- **Update** — when a concept changes, propagate the change. You tell the wiki "from now on we think about X this way" and the agent figures out which pages need editing.

The structure on disk is straightforward: a `SCHEMA.md` at the root that captures conventions, a `raw/` directory with the immutable source documents you've fed in, a `wiki/` directory with `index.md`, `log.md` (an append-only operation log), `overview.md` (an evolving synthesis), and a flat `pages/` directory with one slug-named markdown file per page. The flatness matters — there's a hierarchical variant but flat is what the skill we used assumes.

The skill we used is [`wiki-skills`](https://github.com/kfchou/wiki-skills), a fork and adaptation of Karpathy's [original gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f). It has a `wiki:init` step that interrogates you about your purpose and helps lay out the schema, and then ingest/query/lint/update skills for the four operations.

Trade-offs versus the other approaches: speed is medium — sometimes slower than RAG, sometimes faster than pure agentic search because the index helps the agent navigate. Performance on the questions it's designed to answer is very good. Complexity is low — the concept is simple, the storage is plain markdown, and you can hold the whole thing in your head. The two real costs are scalability and manual work.

Scalability tops out somewhere in the hundreds of pages. Past that, the agent struggles to keep the structure coherent during edits. You can extend the ceiling significantly by combining the wiki with RAG — the wiki becomes the human-readable layer, and `qmd` or similar provides fast retrieval over both the wiki pages and the raw sources behind them. That pushes the ceiling toward thousands, at the cost of bringing all of RAG's complexity back into the picture.

Manual work is the bigger one, and it's the price of the human-readable property. After every ingest, you have to review the wiki edits. Because the wiki becomes the single source of truth for your domain, anything that goes wrong here will propagate to code generation, help portal output, architecture decisions, anything downstream. A bad page poisons everything that reads from it. In a team setting we've been considering a rule that wiki changes need three peer reviews before merging — like a heavily-protected branch — because the cost of error compounds.

Crucially, the wiki always has the raw sources behind it. The `raw/` directory keeps the immutable originals. If you decide your wiki has drifted or compounded errors, you can rebuild from raw at any time. Every page also tracks where its content came from — this fact came from that meeting note, that fact came from this PDF — so you can always trace back. There's a changelog as well, recording what changed in the wiki and why.

A participant who'd been experimenting with this for a while shared a real observation: building the wiki from a codebase works, but it's incomplete. He had pulled in the editor repositories and asked the agent to build wiki pages about the template push/pull/update flows. The push flow was discovered cleanly. The update flow was missed entirely, and he only noticed because he happened to know it existed. Asking the agent directly about "pull" surfaced it, but if he hadn't known to ask, that knowledge would have been silently absent. This is the limit of code-as-documentation: the code expresses what the system does, but not everything important is explicit there. The fix is to feed in additional sources — help portal pages, architecture documents, decision records — alongside the code, so the wiki gets the full picture.

A participant also pointed out that the wiki agent doesn't always maintain the index perfectly. Links break occasionally, references go stale. Not catastrophically, but enough that the lint step matters and you can't trust the wiki to stay healthy without occasional human attention.

There was a broader question that came up: what happens when we have multiple wikis? An editor wiki, a personalisation wiki, a segmentation wiki — each is reasonable on its own, but they overlap, and the cross-product is where most of the interesting questions live. One option is to give each domain its own wiki and let the agent decide at query time which one to enter. The problem is that the wiki's strength comes from the cross-links — if the editor wiki doesn't link to the personalisation wiki, the agent that picks "editor" at the top level will never find a personalisation answer, no matter how deep it goes. The cross-links have to exist between wikis, and that's a hard structural problem we haven't solved. Another option is to put a domain-specialist agent in front of each wiki and have an orchestrator agent route questions. That adds layers, and every layer is somewhere quality can drop, but it might be necessary. We'll probably try several variants in the coming cycles and see which holds up.

Another participant noted that not everything actually belongs in a wiki. Architecture is one example — a diagram captures the structure of a system far more densely than a paragraph of wiki prose can, and the agent can read the diagram. Behavioural specifications might live as Gherkin files in a repository rather than as wiki pages, because Gherkin is human-readable and the agent can grep it. The wiki is for concepts and high-level structure; other specifications live alongside it. A query then becomes a hybrid: the agent looks at the wiki for concepts and at the Gherkin files for behaviour, and the combination is more useful than either alone.

The hands-on portion used a folder of deep-research outputs about diets and Hungarian agriculture as source material. The exercise was: `wiki:init` to scaffold the wiki, ingest the deep-research documents, ingest some additional recipes, then run an update to apply a conceptual change. Several people noticed that when they asked Claude to ingest a whole folder, it sometimes bypassed the skill entirely and parsed files manually. The skill expects you to call `wiki:ingest` per document; calling it on a folder, or describing the task in general terms, can lead the agent to do its own thing. Starting a session with the skill explicitly (`/wiki:ingest …`) is the most reliable way to ensure it's used, and if you want to be sure, you can ask the agent in-session whether it used the skill — it won't lie about it.

## Knowledge Engine

The fourth approach is the newest and the most speculative — the tooling barely exists yet, and what does exist is mostly inside one or two vendor products. The shape is interesting enough to talk through even so.

The core idea: a knowledge engine processes raw inputs into use-case-specific structured stores, and provides a use-case-specific query interface on top. Where RAG vectorises everything uniformly and agentic search searches everything raw, the knowledge engine extracts the data each use case actually needs into the shape that use case needs.

A recipe example makes this concrete. Suppose I have a folder of recipe documents in mixed formats — some are prose, some are markdown tables, some are scraped HTML. The use cases I care about are "what can I cook in under 20 minutes?" and "what's keto-friendly?" and "what uses ingredients I already have?" For each of those use cases I write a parser that pulls structured data out of the raw recipe — calories, prep time, ingredients, dietary tags — and stores it in a typed store. Queries hit the structured store, not the raw documents, so they return in milliseconds with the right structure.

A knowledge engine use case has two parts: a parser that extracts structured data from raw input, and a query interface that runs against the parser's output. The parser is where the smartness lives. The interesting thing is that vendors building this category are pushing toward agents that build the parsers and the queries for you. You give the system a set of evaluation pairs — questions with the expected answers — plus a library of skills (scripts with descriptions) and let an agent iterate. It generates a parser, generates query logic, runs the evaluations, sees what fails, adjusts. Eventually the evaluations pass and you have a working use case you didn't have to design.

The classification logic inside the parser can itself be a small trained model. The agent decides whether the data is structured enough to use deterministic parsing, or whether it needs to invoke an LLM, or whether it should train a small classifier and use that. The whole thing turns into a black-box "I give you examples, you give me a fast structured query interface" pipeline.

Pinecone has been pushing in this direction with their newer product. They've built a custom query language on top of their database that handles the query side of this; the agent emits queries in their JSON-shaped DSL rather than SQL. The premise of selling the database is that all of this is hard enough that you'd rather pay for the abstraction than build it yourself.

You can absolutely do this with off-the-shelf parts. The version I built for myself uses SQLite. Meeting transcripts are the use case — I extract action items, summaries, decisions, attendees, and a few other things, store them as JSON conforming to a schema, load them into SQLite, and query with regular SQL. The questions I actually ask of my own meetings ("what are my action items?", "when did we last discuss X?") return instantly because the data is in the right shape. The schema is the contract: the parser knows what to fill in, the queries know what to ask for.

Trade-offs: speed is excellent for the cases the schema covers, because at query time you're running SQL against indexed structured data. Complexity is high — every use case is its own parser, its own schema, its own queries, though most of the components reuse the same patterns. Performance is excellent within the use cases you've built, and unhelpful outside them. Scalability is good — far better than the wiki, because structured storage handles thousands or tens of thousands of rows without thinking. Manual work is low after the initial setup: once a parser exists, ingesting a new document is automatic, and the structure means the agent doesn't have to interpret the data each time it queries.

Human-readability is "possible" — the JSON or the SQL tables are inspectable, but they're not as immediately scannable as a wiki page. You can build a notebook-style summary on top if you need humans to read it, but the primary consumer is the agent.

There's a subtle but important conceptual point. RAG embeds everything uniformly; the wiki compiles everything into one uniform format (markdown pages). The knowledge engine deliberately doesn't unify — it makes the storage shape match the use case. That's an architectural pendulum swing. We've been moving toward universal representations and now we're moving back toward specific ones. I suspect the next wave will swing the other way again toward something more standardised, but for now use-case-specific structured storage is where the highest-quality systems are.

A participant raised a real concern about this. If every team builds use-case-specific stores, we lose the property that one team's questions can be answered by another team's data — the editor team's structured store doesn't know anything about personalisation, and the personalisation team's store doesn't know anything about the editor. The wiki at least has cross-linkable pages; the structured stores don't. The honest answer is that we don't know yet how to solve this cleanly. We'll likely end up with company-level shared knowledge (the things everyone needs) plus personal or team-level specific stores, and figuring out the boundaries between them will be one of the hard problems in this cycle. We also have to decide what goes in the wiki versus a structured store: architecture probably lives as diagrams and OpenAPI specs rather than as wiki pages, behavioural specs probably live as Gherkin, recipe-like things live as structured stores, and the wiki holds the conceptual glue.

The hands-on portion of the workshop was a build-your-own knowledge engine for the recipe corpus. The exercise: given the folder of recipes in mixed formats, write parsers that extract structured data, choose a storage shape, and expose a query interface. There was no skill ready to drop in — the technology mostly doesn't exist as off-the-shelf tooling yet — so each person had to design it from scratch.

The solutions diverged in interesting ways. One participant landed on a clean architecture: agent extracts each recipe to JSON conforming to a schema, JSON files load into DuckDB, an MCP server exposes a `query_recipes` tool that runs DuckDB queries. The MCP server had the query shapes hard-coded — selecting specific fields and matching on tags. Pulled back a level, this is exactly the knowledge-engine shape: parse raw documents into a structured store, expose a query interface. The difference from the vendor versions is that the vendor lets the agent write arbitrary queries against a generic schema. Pushing the participant's design that way meant exposing the DuckDB schema to the agent and letting it write SQL — at which point you've built the knowledge engine.

Another participant let Claude write extraction Python scripts, with the Council skill consulted for sanity checks. The Council insisted the simplest workable approach — one parser, one big JSON, one query path — was the right one, and the participant let it stand. The schemas the agent produced were good: rich field descriptions, validation rules, enums for known values. Those descriptions matter more than they look, because they're what the agent reads when figuring out which field maps to a query. A schema where every field has a one-line description outperforms a bare schema by a noticeable margin.

A third participant let the agent iterate through regex-based extraction. It worked but the script ballooned past 800 lines as the agent added special cases for each format. That illustrates the parser-quality trade-off: with a fixed input format, deterministic parsing is fast and cheap. With variable inputs, you either invest in a single very smart parser (LLM-based extraction with a schema and validation tools) or you ship an ever-growing pile of heuristics that breaks the next time a new format arrives. The LLM-extraction route is more expensive per document but scales to new formats without code changes.

One observation I'd add: feeding the agent a JSON schema with rich descriptions for each field is dramatically better than letting it figure out the structure on its own. The schema gives it a target. With descriptions, you can validate the output and rerun on validation failure, which catches most edge cases automatically. Without descriptions, the agent improvises field names and you end up with inconsistent data.

Use-case definition was the thing people skipped first and noticed later. The whole premise of the knowledge engine is that the schema and query shapes are derived from the use cases — what questions are you going to ask? Several people built a generic recipe schema first and only afterwards discovered which queries they actually wanted, which is the wrong order. The vendor versions of this technology lean hard on the use case definition: you supply input/output examples, the agent infers the schema and the queries from those. We'll come back to this in a dedicated session — build the input/output evaluation pairs first, let the agent figure out the structure.

## Comparing the four

The summary table from the slides is worth reproducing because it makes the trade-offs sharp:

| | RAG | Agentic Search | Karpathy Wiki | Knowledge Engine |
|---|---|---|---|---|
| **Speed** | Fast | Slow | Medium | Fast |
| **Complexity** | High | Medium | Low | Medium |
| **Performance** | Good | Best | Great | Best |
| **Human Readable** | No | No | Yes | Possible |
| **Flexibility** | Low | High | Medium | Medium |
| **Scalability** | 100K+ docs | ~Infinite | Hundreds | Thousands |
| **Manual Work** | Small | Small | High | Small |

Each has a sweet spot. RAG when you have many documents and need fast retrieval. Agentic search when you need maximum flexibility and can spend the wall-clock time. Wiki when you want a knowledge representation humans can read and audit, at small to medium scale. Knowledge engine when you have clear, repeatable use cases and want fast structured answers.

The point isn't to pick one. The point is that the right system layers them: a wiki holds the conceptual model, a knowledge engine answers structured questions about specific domains, RAG provides fast keyword and semantic retrieval over the long tail, and an agent on top decides which layer to consult for a given question. Each layer covers what the others can't.

We're at the awkward middle of this evolution. The wiki tooling exists and works. The knowledge engine tooling barely exists outside vendor products. RAG tooling is finally good. Agentic search is built into every agent framework already. Knowing the shape of each helps you decide what to build now and what to wait for.

## A tangent that may matter more than the rest

Mid-way through the knowledge engine discussion, someone raised a question that pulled the workshop sideways. Why are we trying so hard to extract structured information from meeting transcripts? The transcript is a low-quality source — it's a long ramble we have to parse, deduplicate, and clean. What if the meeting's deliverable was a structured document in the first place?

The straw-man version: kill the meeting. Replace it with a document. Everyone writes their point in the appropriate field, the document is the structured input the agent needs, and the human review built into the writing process means we never feed slop into the knowledge base.

The objection: meetings aren't just data collection, they're discussion. A conversation that goes back and forth between two people can reach a third idea neither of them would have written down. You can't replace that with a form.

But also: an enormous number of our meetings aren't discussions. They're status updates, decision reviews, async-able coordination. The standup that summarises what each person did yesterday could be a Confluence page where everyone writes their three lines and reads everyone else's. The decision review could be a document with structured fields for context, options, criteria, and individual votes — exactly like the decision documents we already write, but with the review actually happening on the document instead of in a meeting where two people argue and everyone else half-listens. The townhall could be an email.

A participant brought up Google Wave. The technology was right and the moment was wrong; we may be back in a moment where the technology fits. Wave was supposed to replace email and didn't, but a Wave-shaped tool — a structured document where multiple people contribute, an LLM keeps the synthesis up to date, comments and threads aggregate around specific claims, and the final state is something both humans and agents can consume — looks like exactly the thing this knowledge-base evolution is converging toward.

The "Council" pattern is also relevant here. In meetings, conversation is serial — one person speaks, the rest wait, the loudest voices dominate. A round-based structured document where everyone has to fill in their position, and only after that does anyone see anyone else's, surfaces the quiet voices and reduces groupthink. The mechanism we're trying to recreate (everyone contributes, then everyone integrates) is exactly the Council mechanic, applied to people.

I'm not naïve about this — getting people to fill in a document async is harder than getting them to show up to a meeting, because the meeting is itself a forcing function. People don't want to read; they want to be present. A participant pointed out that just having the document doesn't ensure people read it; the meeting is an enforcement mechanism, an expensive one but a real one. Another suggested a hybrid: schedule the meeting slot, but spend it filling in and commenting on the structured document together. You get the forcing function of the meeting and the structured output of the document.

For onboarding-style content where context is needed, a NotebookLM-style podcast generated from the document gives you the conversational flow some people learn from, without requiring an actual conversation. The participant who'd raised the discussion-isn't-just-data point conceded that this hybrid covered the main thing meetings give you.

The case where this is most obviously a win is decision documents, which we already write before contentious meetings. Right now they speed the meeting up — everyone arrives with the same shared context. The next step is to run the actual decision asynchronously on the document. First round: everyone reads it, asks clarifying questions, the doc is improved. Second round: everyone votes and writes their reasoning. Third round, if needed: discussion in the comments. A meeting at the end if there's still disagreement. The whole workflow could compress dramatically.

Whether or not we ever build a product around this, the team can experiment with the workflow internally. The pattern fits our knowledge-base evolution: structured human input flows into a knowledge engine, the engine maintains the wiki, agents query across all of it. The meetings we don't need to have were never the point — the structured knowledge artifacts they were trying to produce were.

We'll probably look at moving toward this in the next workshop, starting with the meetings most obviously meeting-shaped and seeing how much can move to documents without losing what made the meetings useful.
